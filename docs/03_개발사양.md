# 언더그라운드맵 — 개발 사양 (v0.2)

> 대상: MVP 구현자. 결정 사항은 사용자 확정에 따름 — **정적 HTML + Tailwind 프론트 · Supabase 백엔드 · GitHub 배포**.
>
> **v0.2 변경점**
> 1. 게이팅 단위 `cities` → `regions`로 일반화. **서울=자치구(구), 그 외=시.** (`metro`=광역 그룹, `kind`='gu'|'si')
> 2. **지역 이용권(`access_passes`)** 추가. 홈 지역이 아닌 지역을 한시 열람·참여하는 유료 패스. RLS 게이팅의 유일한 예외이자 새 수익 라인.
> 3. 모든 `*.city_id` → `*.region_id`. 헬퍼 `my_open_city()` → `my_home_region()` + `can_view_region()`.

## 1. 기술 스택 (확정)

| 레이어 | 선택 | 이유 |
|---|---|---|
| 마크업/스타일 | **정적 HTML + Tailwind (CDN)** + 소량 커스텀 CSS | 빌드 없음, 비개발자 수정 쉬움, 기존 산출물과 통일 |
| 폰트 | Pretendard (CDN) | 한글 가독성·기존 통일 |
| 로직 | **Vanilla JS (ES Modules)** | 프레임워크 불필요, 화면 스위칭 SPA |
| 백엔드 | **Supabase** (Postgres + Auth + RLS) | 인증·DB·행 보안 한 번에, anon key 클라이언트 노출 안전 |
| 배포 | **GitHub Pages** (또는 Netlify/Vercel) | `git push` = 배포 |
| 광고 | 좌우 레일 슬롯 (AdSense-ready placeholder) | 기획 요구 |

> **빌드 스텝 없음.** `index.html`을 그대로 서빙. Node/번들러 불필요.

## 3. 앱 셸 레이아웃 — 중앙 컬럼 + 광고 레일

**요구:** 앱 UI. 양쪽 비고, 가운데 핸드폰 너비 한 줄(스퀘어테이블식). 좌우 여백 = 광고.

```
┌───────────────────────────────────────────────────────┐
│  data-ad-left       [ 앱 셸 (max 480px) ]     data-ad-right │
│  (≥1100px에서만)    ┌───────────────────┐   (≥1100px에서만)  │
│  ┌────────────┐    │  상단 앱바          │    ┌────────────┐  │
│  │            │    │───────────────────│    │            │  │
│  │  광고 300  │    │  화면 컨텐츠(스크롤) │    │  광고 300  │  │
│  │  x 600     │    │                   │    │  x 600     │  │
│  │            │    │───────────────────│    │            │  │
│  └────────────┘    │  하단 탭바          │    └────────────┘  │
│                    └───────────────────┘                    │
└───────────────────────────────────────────────────────┘
```

- 컨테이너: `display:flex; justify-content:center; gap`. 중앙 `main.app-shell { width:100%; max-width:480px }`.
- 앱 셸은 뷰포트 높이 고정(`100dvh`), 내부 컨텐츠만 스크롤 → 앱 감.
- **반응형 브레이크포인트**
  - `< 1100px`: 광고 레일 **숨김**(`display:none`). 중앙 컬럼만.
  - `≥ 1100px`: 좌우 레일 노출(각 300px 권장, 최소 160px).
  - `< 480px`(모바일): 중앙 컬럼이 화면 100% 차지.
- **광고 슬롯 규격**: 데스크톱 사이드 = 반응형 세로 배너(권장 300×600, 대체 160×600). 슬롯은 `assets/js/ads.js`가 채움. 초기엔 자체 하우스 배너(팀 소개/제휴 매장 모집) → 이후 AdSense `<ins class="adsbygoogle">` 또는 직접 판매 배너로 교체. **인증 게이트(S0–S2) 전에는 광고 미노출**(첫인상 보호), 피드 이후 노출 권장.

## 5. 데이터 모델 (요약 — 상세는 `supabase/schema.sql`)

| 테이블 | 핵심 컬럼 | 비고 |
|---|---|---|
| `regions` | id, name, slug, **metro**, **kind('gu'\|'si')** | 게이팅 단위. 서울=구, 그 외=시 |
| `profiles` | id(=auth.uid), **region_id**, display_name, resident_verified, is_host, is_venue_owner | auth.users 1:1. region_id=홈 지역 |
| `venues` | id, owner_id, **region_id**, name, address, category, capacity, idle_days, idle_start, idle_end, slot_minutes | 공간 공급(홈 지역) |
| `meetups` | id, host_id, venue_id, **region_id**, title, description, category, starts_at, duration_min, capacity, fee, **venue_share_pct**, status | 참가비·**매장분배율 필수** |
| `registrations` | id, meetup_id, user_id, status, created_at | 신청. (meetup_id,user_id) unique |
| `access_passes` | id, user_id, **region_id**, kind('guest'\|'period'), meetup_id, price, issued_by, valid_from, **valid_until**, status | **지역 이용권**. 크로스-리전 열람·참여 |

**불변식(스키마·앱 공통)**
- `regions.kind` = 'gu'(서울 자치구) | 'si'(그 외 시). 서울만 구 단위로 세분.
- `meetups.venue_id` NOT NULL (공간 종속).
- `meetups.fee >= 0`, `venue_share_pct` 0–100 (분배 필수 개념 보존).
- `meetups.region_id`는 트리거로 `venues.region_id`와 일치 강제(지역 불일치 모임 방지).
- `access_passes.valid_until > valid_from`, `status in ('active','expired','revoked')`. **홈 지역엔 발급 불가**(이미 열려 있음).

## 6. 지역 단위 게이팅 — RLS 전략 (핵심)

게이팅은 **프론트 숨김이 아니라 Row Level Security로 DB에서 차단**한다. 단위 = **서울은 구, 그 외는 시.**

**헬퍼 함수(security definer)**
- `my_home_region()` → 거주 인증된 경우에만 `profiles.region_id`, 아니면 NULL.
- `can_view_region(rid)` → `rid = my_home_region()` **또는** 그 지역의 **유효한 이용권**(`access_passes.status='active' and valid_until > now()`) 보유 시 true.

**정책 원칙**
- `venues` / `meetups` `SELECT`: `can_view_region(region_id)` → 홈 지역 **또는 이용권 보유 지역**만 반환.
- `venues` `INSERT`: `owner_id = auth.uid()` + `is_venue_owner` + **`region_id = my_home_region()`**(공급은 홈 지역만, 방문자 불가).
- `meetups` `INSERT`: `host_id = auth.uid()` + `is_host` + **선택 venue가 홈 지역**.
- `registrations` `INSERT`: 본인(`user_id=auth.uid()`) + 대상 meetup이 `status='open'` + **`can_view_region(meetup.region_id)`**(홈 또는 이용권 방문자 모두 참여 가능).
- `access_passes`: 본인 것만 `SELECT`. `INSERT`(구매)는 `user_id=auth.uid()` + **`region_id ≠ my_home_region()`**.
- `profiles`: 본인 행만 select/update. 가입 시 insert 본인 한정.

> 결과: 인증 안 된 사용자·이용권 없는 타 지역 사용자는 쿼리해도 **빈 결과**. "관광객에게 보이지 않는 지도"가 데이터 레벨에서 성립하고, **이용권만이 그 벽을 넘는다.** 정확한 정책 SQL은 `supabase/schema.sql` 참조.

## 6-1. 지역 이용권(크로스-리전 패스) — 상세

- **개념:** 홈 지역이 아닌 지역을 **한시적으로 열람·참여**하는 유료 패스. 소지자는 대상 지역에서 **방문자(visitor)** 상태가 된다.
- **권한 범위:** 열람 O · 모임 참여 O · **호스팅 X · 매장등록 X**(공급 무결성은 홈 지역 거주자만). → 로컬 커뮤니티의 신뢰·밀도를 방문자가 희석하지 않는다.
- **상품(플랫폼 판매, `data.js:PASS_PRODUCTS`):**
  | key | kind | 기간 | 가격(예시) |
  |---|---|---|---|
  | `period-30` | period | 30일 | ₩9,900 |
  | `period-90` | period | 90일 | ₩24,900 |
  | `guest-7` | guest | 7일 | ₩4,900 |
- **수익:** 이용권 결제액의 플랫폼 수수료(초기엔 전액을 플랫폼 수익으로 잡는 단순 모델, 추후 지역 커뮤니티 환원 비율 설계). 수익 모델 §은 확정기획 §10 참조.
- **발급 흐름(MVP):** `passes` 화면에서 지역+상품 선택 → `db.buyPass()` → `access_passes` insert(status='active', valid_until=now+기간) → 피드가 그 지역으로 전환. *결제 PG는 후순위(발급 시뮬레이션).*
- **만료:** `valid_until` 경과분은 `can_view_region`이 자동 제외. (배치로 `status='expired'` 마킹은 선택.)
- **백로그(G4):** 호스트가 특정 모임에 **초대권(guest, meetup_id 지정)을 발행·판매** → `issued_by`=호스트, 플랫폼 수수료. 2차 마켓/기업·이주자 번들도 여기서 확장.

## 7. 인증 플로우 (Supabase Auth)

1. **가입/로그인**: 이메일 매직링크(기본) 또는 전화 OTP. MVP 기본값 = 이메일 매직링크(설정 간단).
2. 최초 로그인 후 `profiles` 없으면 생성(지역 선택 화면 S2).
3. **거주 인증(간이)**: **지역 선택(서울=구·그외=시, `<optgroup>`)** + 자기신고 체크 → `profiles.resident_verified=true`, `region_id` 저장.
   - *주의: MVP는 자기신고. 실제 주소·행정 인증은 후순위(기능정의 §7).*
4. 세션은 Supabase가 localStorage에 보관. 앱 진입 시 `getSession()`으로 복원 → 라우팅 가드.

## 8. 설정 / 시크릿

- `config.example.js`:
  ```js
  window.UG_CONFIG = {
    SUPABASE_URL: "https://YOUR-PROJECT.supabase.co",
    SUPABASE_ANON_KEY: "YOUR-ANON-KEY"
  };
  ```
- 사용자가 `config.js`로 복사 후 값 입력. **anon key는 공개돼도 안전**(RLS가 보호). service_role 키는 **절대 클라이언트에 넣지 않는다.**
- `config.js` 미존재/미설정 시 앱은 **mock 모드**(seed 데이터, 저장은 메모리)로 동작 → GitHub Pages에서 키 없이도 데모 가능.
- `.gitignore`에 `config.js` 포함(공개 저장소 시 키 노출 방지). *GitHub Pages는 정적이라 서버 env가 없으므로, 배포본에 키를 넣으려면 config.js를 배포 브랜치에 포함하거나 CI로 주입.*

## 9. 배포 (GitHub Pages)

1. `언더그라운드맵/`를 저장소 루트로 하는 리포 생성 (또는 이 폴더를 리포로 init).
2. `main` 브랜치 push.
3. GitHub → Settings → Pages → Source: `main` / root → 저장.
4. `https://<user>.github.io/<repo>/` 로 공개.
5. Supabase 대시보드 → Authentication → URL Configuration에 **배포 URL을 Redirect/Site URL로 등록**(매직링크 콜백).
6. (선택) 커스텀 도메인·Netlify/Vercel 사용 시 env로 키 주입 가능.

**로컬 실행**: 정적이지만 ES Module·fetch 때문에 `file://` 대신 로컬 서버 필요.
```
# 폴더에서
python -m http.server 5173      # → http://localhost:5173
```
